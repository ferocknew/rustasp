# 设计缺陷文档

## 问题：语句解析器死循环

### 发现日期
2025-03-05

### 严重程度
🔴 **严重** - 导致解析器死循环，CPU 100%，ASP 页面无法正常响应

### 问题描述

当 ASP 页面包含单行 If 语句时（如 `<%if x=1 then Response.Write("Yes")%>`），解析器会进入死循环，导致：
- CPU 使用率飙升至 100%
- 页面请求超时
- 服务器无法正常响应

### 根本原因

语句解析器 `StmtParser` 使用了**错误的解析模式**：

```rust
// src/parser/stmt_parser/core.rs:101-148
fn parse_assignment_or_expr(&mut self) -> Result<Option<Stmt>, ParseError> {
    // ❌ 错误模式：复制 tokens
    let mut tokens = vec![];
    while !self.is_at_end() && !self.is_stmt_end() {
        tokens.push(self.advance().clone());  // 消耗原始 tokens
    }

    // 调用表达式解析器处理复制的 tokens
    let expr = parse_expression(tokens)?;

    // ⚠️ 问题：原始位置 self.pos 已前进
    // 但 parse_expression() 在新数组上操作，不影响 self.pos
}
```

**致命缺陷**：
1. `parse_assignment_or_expr` 消耗原始 tokens（位置前进）
2. 但 `parse_expression` 在**复制的 tokens 数组**上操作
3. `parse_program` 主循环检测到位置前进，继续下一轮
4. 下一轮调用 `parse_assignment_or_expr`，发现已到末尾，返回 `Ok(None)`
5. **但主循环继续**，因为没有语句被添加，位置未改变
6. 无限循环

### 复现路径

1. 访问包含单行 If 的 ASP 页面：
   ```asp
   <%
   dim x
   x = 1
   if x = 1 then Response.Write("Yes")
   %>
   ```

2. 解析流程：
   - `parse_program` 主循环开始
   - 调用 `parse_stmt()` → `parse_if()`
   - `parse_if` 调用 `parse_stmt()` → `parse_assignment_or_expr()`
   - `parse_assignment_or_expr` 消耗 tokens 返回语句
   - **下一轮迭代，`parse_if` 循环继续**
   - 再次调用 `parse_stmt()`，位置未前进
   - **死循环**

### 当前临时方案

已添加循环计数器和位置检查作为兜底：

```rust
// src/parser/stmt_parser/core.rs:32-48
if loop_count > 100 {
    return Err(ParseError::ParserError(
        "解析程序时检测到可能的死循环"
    ));
}

// 强制前进避免死循环
if self.pos == last_pos && loop_count > 1 {
    self.advance();  // ⚠️ 临时方案：强制前进
}
```

**临时方案效果**：
- ✅ 避免真正的死循环
- ⚠️ 会跳过 token，导致解析不完整
- ⚠️ 可能产生错误的 AST

### 正确的解决方案

需要重构语句解析器，有以下两种方案：

#### 方案 A：完全移除 token 复制（推荐）

```rust
// 让 parse_assignment_or_expr 直接调用表达式解析器
// 而不是复制 tokens

fn parse_assignment_or_expr(&mut self) -> Result<Option<Stmt>, ParseError> {
    // 直接使用表达式解析器，不复制 tokens
    let start = self.pos;

    // ... 解析逻辑，直接消耗 tokens

    let end = self.pos;

    // 验证位置是否前进
    if end == start {
        return Ok(None);  // 没有解析到任何内容
    }

    Ok(Some(stmt))
}
```

#### 方案 B：重构为标准递归下降解析器

完全移除 `parse_assignment_or_expr`，在 `parse_stmt` 中直接处理所有语句类型：

```rust
fn parse_stmt(&mut self) -> Result<Option<Stmt>, ParseError> {
    match self.peek()? {
        // ... 各种关键字

        // 直接处理赋值语句
        Token::Ident(name) => {
            // 检查下一个 token 是否是 =
            if self.check_ahead(Token::Eq) {
                return self.parse_assignment();
            }
            return self.parse_expr_statement();
        }

        // ... 其他情况
    }
}
```

## ✅ 修复完成

### 修复日期

### 修复方案
采用**工业级模块化架构重构**，完全移除 token 复制逻辑：

#### 新架构
```
parser/
├── mod.rs          // 模块导出
├── parser.rs       // Parser struct + 基础工具
├── program.rs      // parse_program
├── expr/           // 表达式解析（Pratt 算法）
│   ├── pratt.rs    // Pratt 主入口
│   ├── prefix.rs   // 前缀表达式
│   ├── infix.rs    // 中缀运算符
│   └── postfix.rs  // 后缀表达式
└── stmt/           // 语句解析（递归下降）
    ├── core.rs     // parse_stmt 分发
    ├── if_stmt.rs
    ├── loop_stmt.rs    // for/while/do
    ├── decl_stmt.rs    // dim/const
    ├── assign_stmt.rs  // 赋值
    ├── proc_stmt.rs    // function/sub
    └── select_stmt.rs
```

#### 核心改进

1. **一个 Parser struct** - 所有解析共享 `self.pos`
2. **通过 impl 分文件扩展** - 每个文件都是 `impl Parser`
3. **单行 If 正确处理** - 检测换行符，单行只解析一次语句
4. **无 token 复制** - 所有解析直接操作 `self.pos`

#### 关键代码
```rust
// parser.rs - 只放基础工具
pub struct Parser {
    tokens: Vec<Token>,
    pos: usize,
}

// stmt/if_stmt.rs - 单行 If 处理
pub fn parse_if(&mut self) -> Result<Option<Stmt>, ParseError> {
    self.expect_keyword(Keyword::If)?;
    let cond = self.parse_expr(0)?;
    self.skip_newlines();
    self.expect_keyword(Keyword::Then)?;
    self.skip_newlines();

    // 检查是单行 If 还是多行 If
    let is_multiline = self.check(&Token::Newline)
        || self.check_keyword(Keyword::End)
        || self.check_keyword(Keyword::Else);

    if !is_multiline && !self.is_at_end() {
        // 单行 If - 只解析一条语句
        let stmt = self.parse_stmt()?;
        let body = stmt.map_or(vec![], |s| vec![s]);
        // ...
    }
    // ...
}
```
