# ASP 父路径安全处理方案

Classic ASP 的父路径 (`..`) 是历史遗留问题，可能导致目录遍历和安全漏洞。

本文档简明总结在 Rust ASP Runtime 中的处理方案。

---

## 1. 问题概述

* 旧 ASP 允许 `../` 访问上级目录
* 攻击者可能读取 WebRoot 外文件
* `Server.MapPath` 和 `#include file` 都可能受影响

---

## 2. 安全原则

1. 所有路径必须落在 WebRoot 内
2. 默认禁止父路径，必须显式开启兼容模式
3. 所有路径解析统一使用安全入口

---

## 3. Rust 安全实现示例

```rust
use std::path::{Path, PathBuf};

fn resolve_path(
    web_root: &Path,
    current_file: &Path,
    input: &str,
) -> Result<PathBuf, RuntimeError> {
    let base_dir = current_file.parent().unwrap();
    let joined = base_dir.join(input);
    let canonical = joined.canonicalize()?;
    if !canonical.starts_with(web_root) {
        return Err(RuntimeError::SecurityViolation);
    }
    Ok(canonical)
}
```

* `canonicalize()` 解析绝对路径
* `starts_with(web_root)` 确保不越界

---

## 4. 模式分级

| 模式     | 描述                     |
| ------ | ---------------------- |
| Strict | 禁止父路径，老代码需修改           |
| Safe   | 允许 `..`，但必须在 WebRoot 内 |
| Legacy | 完全允许父路径，仅用于测试，不推荐      |

---

## 5. include 与 MapPath

* `file` 相对当前文件路径
* `virtual` 相对 WebRoot 路径
* Server.MapPath 使用同一 resolve_path 入口

---

## 6. 实现建议

* 建立统一 `path_resolver.rs` 模块
* include / MapPath / 静态文件加载共用入口
* 默认沙箱化，兼容老系统可选开关

---

## 7. 总结

父路径问题不是语法问题，而是**运行时文件系统沙箱设计问题**。

核心策略：**统一解析 + canonicalize + 限制在 WebRoot + 提供模式开关**。
