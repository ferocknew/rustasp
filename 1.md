# asp-lite 项目目录结构说明（Rust 实现 Classic ASP 子集）

本项目目标：

* 使用 **Rust** 实现一个 Classic ASP（VBScript 子集）运行时
* 摆脱 IIS，支持容器化部署
* 不支持 COM / ActiveX / Windows-only DLL
* Access 迁移为 SQLite
* 支持 Nginx 反向代理
* 强调可维护性、可扩展性、模块解耦

---

# 一、整体架构分层

系统采用分层设计：

```
HTTP 层
  ↓
ASP 引擎层
  ↓
VBScript Runtime（解释执行层）
  ↓
Parser / AST（语法层）
```

各层职责清晰，避免强耦合。

# 三、各模块职责说明

---

## 1️⃣ ast/ —— 抽象语法树层

作用：

* 定义 VBScript 的语法结构
* 只包含数据结构（enum / struct）
* 不包含执行逻辑

包含内容：

* 表达式（Expr）
* 语句（Stmt）
* 运算符（BinaryOp）
* Program 结构

为什么单独拆？

* 方便做静态分析
* 方便做代码自动修复
* 解耦 parser 和 runtime

---

## 2️⃣ parser/ —— 语法解析层

作用：

* 词法分析（lexer）
* 语法分析（parser）
* 输出 AST
* 提供错误定位

职责清晰：

输入源码字符串
输出 AST

不执行代码。

拆分原因：

* 表达式解析与语句解析分开
* 防止 parser.rs 超过 500 行

---

## 3️⃣ runtime/ —— 解释执行层（核心）

作用：

* 执行 AST
* 管理变量作用域
* 管理函数调用
* 实现弱类型系统
* 错误控制

这是项目复杂度最高的部分。

---

### runtime/value/ 子目录

专门隔离 VBScript 的弱类型逻辑：

* Value 枚举
* 自动类型转换
* 运算符行为
* 比较逻辑

必须单独拆目录，否则文件会爆炸。

---

## 4️⃣ builtins/ —— ASP 内建对象层

实现：

* Request
* Response
* Server
* Session
* 数据库抽象

重要设计原则：

* 只实现白名单功能
* 不支持自定义 COM
* 不支持 ActiveX
* 不支持 Windows DLL

数据库通过内部抽象接口实现：

* SQLite
* 可扩展 SQL Server

---

## 5️⃣ asp/ —— ASP 引擎层

负责：

* 分割 HTML 与 `<% %>`
* 执行脚本片段
* 拼接输出
* 管理单次请求上下文

不包含 HTTP 逻辑。

这是“模板执行层”。

---

## 6️⃣ http/ —— Web 服务层

负责：

* 路由
* 文件加载
* 构建 Request 上下文
* 返回 HTTP 响应

推荐使用：

* axum

此层不处理 VBScript 逻辑。

# 四、设计原则总结

---

## ✅ 单一职责

每个目录代表一个系统维度：

* 语法
* 执行
* 内建对象
* 模板引擎
* HTTP

避免混在一起。

---

## ✅ 控制文件大小

目标：

* 单文件不超过 500 行
* match 分支过多必须拆文件
* Value 相关逻辑全部集中在 value/ 目录

---

## ✅ 子集策略

不追求完全兼容 VBScript。

支持：

* Dim
* If
* Function
* 基本表达式
* Response.Write

不支持：

* COM
* ActiveX
* Windows-only DLL

---

## ✅ 可扩展性

未来可以：

* 增加静态扫描器
* 增加代码自动迁移工具
* 支持 AOT 编译
* 支持 WASM 运行

# 六、总体定位

本项目本质是：

> 一个安全可控的 Classic ASP 子集运行时
> 一个面向迁移的过渡引擎
> 一个容器友好的 IIS 替代方案

不是 100% 复刻 IIS，而是：

“工程可控的现代化替代实现”。
